7. Open questions, supersede & risks

Design / product checklist

  • Exit criteria — When can legacy duplicate surfaces shrink?
  • Surface ownership — Protect vs legacy for list / detail / QV.
  • QV — “Prod today” vs “target” (CTAs).
  • Supersede design — Full target when legacy retires.
  • Dismissals — Resurface rules (org-wide settled for product; history/notes schema).
  • Noise — False positives before broader rollout.

7 open questions — dupes supersede (engineering)

  1. Data source mismatch — Legacy dupes from ICN (draft → paid/deleted). Protect Horizon events today skew approved bills. Superseding does not cleanly replace coverage.
  2. Non-approved screens — Continue bill-level dupe flagging (shield + top banner) for Draft / Awaiting approval / paid etc., or only Awaiting payment?
  3. Org vs bill-level access — Current UI often fetches all dupes for an org. Bill-level lookup per documentId is a different pattern; BFF/UI refactor scope unknown.
  4. Feature flag — Work to flag new unified Xero Protect view vs legacy.
  5. Dupe detailed view — Does existing comparison/resolution screen stay as-is under a Protect wrapper, or change?
  6. ICN → Horizon — Long-term alignment so Protect can cover the same lifecycle as ICN-backed dupes (migration / dual-read period TBD).
  7. Supersede E2E scope — Front + backend unknowns bundled; depends on resolving 1–5 and legacy ASPX / modern split.

Beta tickets: ADDAPI duplicate bill endpoint; BFF merging shield + duplicates payload; LLM summarisation (effort ?); list banner/action centre; CTA merge on detail/QV (lifecycle-dependent); supersede front+back (?).
